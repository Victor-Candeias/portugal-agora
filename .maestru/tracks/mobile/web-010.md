# WEB-010: Carris: pipeline de build SQLite (WASM) + modelo de dados

## Contexto (revisto)

Baseado em `.Docs/CarrisDev.txt` **e** na proposta do utilizador de arquitetura "SQLite (WASM) no GitHub Pages". `apps/web` é deployado como site 100% estático (`.github/workflows/deploy.yml`, sem servidor, sem BD). A ideia inicial deste WI (backend + BD/cache própria) foi **descartada**: não há nem está previsto servidor no projeto, e criar um exigiria hosting extra fora do GitHub Pages.

**Nova abordagem (validada e mais realista para este projeto):**
- **Dados estáticos** (linhas, paragens, rotas, patterns, shapes, operadores) → gerados **em build/CI** (workflow agendado, ex. diário) num ficheiro `.sqlite` versionado como asset estático, servido pelo GitHub Pages junto com o site.
- No browser, esse ficheiro é carregado com **SQLite compilado para WebAssembly** (ex. `sql.js` ou `wa-sqlite`) e consultado localmente — pesquisas instantâneas, funciona offline após o primeiro carregamento (cache do browser/Service Worker).
- **Dados dinâmicos** (chegadas, veículos, alertas) continuam a ser pedidos em tempo real diretamente às APIs oficiais a partir do cliente, tal como hoje em `useCarris.ts` — não fazem sentido dentro do SQLite estático.

## Objetivo

- Escolher biblioteca SQLite/WASM (`sql.js` vs `wa-sqlite`) — avaliar tamanho do bundle e suporte a `fetch` de ficheiro `.sqlite` por HTTP range/chunks.
- Definir esquema SQLite (tabelas): `operators`, `lines`, `routes`, `patterns`, `stops`, `shapes`.
- Criar módulo `apps/web/src/lib/staticDb.ts` (ou pacote `@portugal-hoje/core`) que carrega o `.sqlite` e expõe queries tipadas, substituindo progressivamente os `fetch` estáticos de `useCarris.ts`.
- Criar workflow GitHub Actions agendado (`schedule:` cron diário) que gera o `.sqlite` a partir da API Carris Metropolitana e o publica como asset do build (`apps/web/public/data/carris.sqlite` ou similar).

## Nota

Esta mesma mecânica (SQLite WASM gerado em CI) é reutilizável no futuro para outros dados estáticos da app (Municípios, Distritos, Freguesias, Pontos de Interesse) — fora do âmbito destes WI, que cobrem apenas Carris.

## Bloqueia

WEB-011, WEB-016 (fases de dados estáticos dependem desta fundação). WEB-013/014/015 (tempo real) passam a depender apenas desta fundação para modelo de tipos comuns, não da geração do SQLite.

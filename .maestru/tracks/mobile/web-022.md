# WEB-022: Metro do Porto: pipeline GTFS->SQLite (WASM) + página de estações/próximas partidas

Verificado em produção após deploy: `https://victor-candeias.github.io/portugal-agora/data/metro-porto.sqlite` responde 200 (~4 MB), gerado pelo workflow `deploy.yml`.

Fonte: `.Docs/metro porto.txt` — arquitetura proposta pelo utilizador: GTFS do Metro do Porto
processado em GitHub Action para um `.sqlite`, publicado como asset estático no GitHub Pages e
consultado no browser via SQLite compilado para WebAssembly (`sql.js`), tal como já implementado
para a Carris (WEB-010/011/016 — mecânica explicitamente pensada para ser reutilizável).

## Fonte de dados GTFS

A URL documentada não existia; foi identificada a fonte oficial real através do portal Open Data
do Porto (CKAN): dataset "Horários, paragens e rotas da Metro do Porto"
(`opendata.porto.digital`, organização `metro-do-porto`, autor `metro@metro-porto.pt`), com
múltiplos recursos GTFS históricos e um rotulado "Mais Recente".

**Nota importante**: o recurso "Mais Recente" (17-07-2026) está publicado com 0 bytes no portal
(testado com `curl`/`fetch` — `Content-Length: 0`). Por isso o script de build
(`apps/web/scripts/build-metro-porto-db.mjs`) não confia no rótulo: percorre os recursos ZIP do
dataset por ordem decrescente de data de criação e usa o primeiro que (a) descarrega com
conteúdo real e (b) contém os ficheiros GTFS essenciais (`routes.txt`, `stops.txt`, `trips.txt`,
`stop_times.txt`), tornando o pipeline resiliente a uploads incompletos futuros do portal.
A estrutura interna do ZIP também varia entre publicações (por vezes os `.txt` estão dentro de
uma subpasta, ex. `Horarios GTFS_09.09.2024/routes.txt`) — a procura de ficheiros é feita por
nome (regex de sufixo de caminho), não por caminho exato.

## Implementação

- `apps/web/scripts/build-metro-porto-db.mjs` — gera `apps/web/public/data/metro-porto.sqlite`
  com o esquema exato proposto no documento: `metro_stations`, `metro_lines`, `metro_trips`,
  `metro_stop_times` (horários convertidos para segundos desde a meia-noite), `metro_calendar`,
  `metro_calendar_dates`, `metro_station_lines` (pré-calculada) + tabela `meta`
  (generated_at/source/gtfs_resource_created). Índices conforme o documento, incluindo o mais
  importante: `metro_stop_times(station_id, departure_seconds)`.
- `apps/web/package.json` — `build:db` passou a correr também este script (reutiliza o mesmo
  workflow agendado diário já existente em `.github/workflows/deploy.yml` para a Carris — não
  foi necessário criar um novo workflow).
- `apps/web/src/lib/metroPortoDb.ts` — carregamento do `.sqlite` via `sql.js`/WASM no browser,
  mesmo padrão de `staticDb.ts` (Carris), mas em ficheiro/instância separados (domínios de dados
  independentes).
- `apps/web/src/hooks/useMetroPorto.ts` — `useMetroStations`, `useMetroStationLines`,
  `useMetroNextDepartures` (calcula os `service_id` válidos para a data atual cruzando
  `metro_calendar` + exceções de `metro_calendar_dates`, tal como recomendado no documento) e
  `useMetroPortoMeta`.
- `apps/web/src/pages/MetroPorto.tsx` — nova página: pesquisa de estações, por estação mostra
  linhas servidas (cores GTFS) e próximas partidas (hora, linha, destino), com mapa opcional
  (reutiliza `SinglePointMap`). Registada em `apps/web/src/main.tsx` (rota `/metro-porto`) e
  `apps/web/src/components/Layout.tsx` (novo item de navegação "Metro do Porto").

## Não implementado (fora deste WI)

- Favoritos/preferências do utilizador (o documento sugere `localStorage`/IndexedDB, separado do
  SQLite estático, só de leitura) — não pedido, fora do âmbito inicial.
- `shapes.txt` (percursos geográficos das linhas no mapa) — o documento sugere evitar se ainda
  não existir mapa de percursos; a app ainda não tem essa funcionalidade para nenhuma linha.
- App mobile (Expo) — esta implementação cobre apenas a app web; o mesmo padrão
  (`metroPortoDb`/`useMetroPorto`) pode ser replicado depois para mobile se necessário.

# WEB-011: Carris: sincronização de dados estáticos (linhas, paragens, rotas, patterns, shapes)

## Contexto

Fases 1–5 do `.Docs/CarrisDev.txt`. Dados que mudam pouco e devem ficar guardados localmente, com sincronização diária.

**Validação:** dados hoje já são obtidos ao vivo em `useCarris.ts` (`useCarrisLines`, `useCarrisStops`, `useCarrisLinePatterns`) só com cache em memória do React Query (staleTime 6–24h) — não há persistência real nem endpoint próprio. Migrar para job de sync diário é razoável e reduz chamadas repetidas à API oficial.

## Objetivo

| Dado | Endpoint origem | Atualização | Destino |
|---|---|---|---|
| Linhas | `GET /v2/lines` | 24h | tabela `lines` no `.sqlite` |
| Paragens | `GET /v2/stops` | 24h | tabela `stops` no `.sqlite` |
| Rotas | `GET /v2/routes` | 24h | tabela `routes` no `.sqlite` |
| Patterns | `GET /v2/patterns/{patternId}` | 24h | tabela `patterns` no `.sqlite` |
| Shapes | `GET /v2/shapes/{shapeId}` | permanente | tabela `shapes` no `.sqlite` |

- Script Node (CI) que faz o download, parse e escreve num ficheiro SQLite via `better-sqlite3` (build-time), publicado como asset estático (não há backend/BD em runtime).
- Módulo client-side (`staticDb.ts`, ver WEB-010) para consultar o `.sqlite` via WASM em vez de repetir `fetch` a cada visita.

# WEB-018: Turismo: integrar SIGTUR/TravelBI (ArcGIS REST) e criar tab/página Turismo

Fonte: `.Docs/Turismo.txt` (SIGTUR/TravelBI via ArcGIS REST Feature Services).

Não existe backend próprio no monorepo — plano adaptado ao padrão client-only já usado
(`packages/core/src/api/dgeg.ts`, `apps/mobile/hooks/useApi.ts`, tabs `apps/mobile/app/(tabs)/*.tsx`,
páginas `apps/web/src/pages/*.tsx`).

## Passos

1. `packages/core/src/api/sigtur.ts` — cliente ArcGIS REST (`query` por layer, paginação via
   `resultOffset`/`resultRecordCount`, bbox e proximidade).
2. `packages/core/src/types/index.ts` — `TourismPointSchema` (zod) com modelo normalizado
   (category, subcategory, coordinates, contacts, source: itemId/layerId/objectId).
3. Configuração inicial só com a layer já validada
   (`SIGTUR_TurismoSaudeBemEstar/FeatureServer/27`); descoberta automática do catálogo ArcGIS
   fica para uma fase posterior.
4. Função de normalização de campos de origem (NOME/DENOMINACAO → name, CONCELHO/MUNICIPIO →
   municipality, etc.).
5. Sem sync diário/BD própria (sem infraestrutura de servidor) — fetch direto do ArcGIS com
   cache via React Query (`staleTime` alto), como em `useFuelPrices`/`useEvStations`.
6. `apps/mobile/hooks/useApi.ts` — `useTourismPoints(bbox|lat/lng, category)`.
7. Tab mobile `apps/mobile/app/(tabs)/turismo.tsx` (padrão de `ev.tsx`).
8. Página web `apps/web/src/pages/Turismo.tsx` com mapa Leaflet (já instalado) e bbox da área
   visível.
9. Registar rota/tab em `apps/mobile/app/(tabs)/_layout.tsx` e no router/`Layout.tsx` do web.
10. Fases futuras (fora deste WI): descoberta automática de layers via ArcGIS Online search e
    sync job diário — exigiria novo serviço backend, ainda inexistente no monorepo.

## Nota de correção (2026-07-17)

O host documentado em `.Docs/Turismo.txt` (`servergeo.sgeconomia.gov.pt`) está inacessível
(TLS connection reset, confirmado dentro e fora do ambiente de dev, incluindo no browser do
utilizador). Via catálogo ArcGIS Online (item `SigTur_Indicadores`, owner `service.desk.tdp`)
identificou-se o host real e funcional: `https://geo.turismodeportugal.pt/server/rest/services/TDP`.
Confirmado com pedidos reais (HTTP 200, dados válidos) à layer
`SIGTUR_TurismoSaudeBemEstar/FeatureServer/27` ("Termas"). O cliente
(`packages/core/src/api/sigtur.ts`) foi atualizado para usar este host e os nomes de campos
reais (`Denominacao`, `Concelho`, `Endereco`, `CodigoPostal`, `Website` em HTML, `Email`),
em vez dos genéricos assumidos no documento original.

## Nota CORS (2026-07-17)

O servidor `geo.turismodeportugal.pt` só devolve `Access-Control-Allow-Origin` para origens
oficiais (ex. `sigtur.turismodeportugal.pt`), bloqueando o browser da app web (CORS). Aplicado
o mesmo padrão já usado para `comboios.live` em `apps/web/src/hooks/useTransportes.ts`:
- `sigtur.ts` (core) expõe `configureSigturRequestUrl(transform)` para reescrever o URL final
  do pedido antes do `fetch`.
- `apps/web/vite.config.ts` — proxy `/api/sigtur` → `geo.turismodeportugal.pt` (dev).
- `apps/web/src/lib/sigtur.ts` — configura o transform: proxy Vite em dev, `corsproxy.io` em
  produção (build estático GitHub Pages, sem servidor próprio).
- Mobile (React Native) não sofre de CORS — usa o host direto sem configuração adicional.

## Ampliação de categorias (2026-07-17)

Adicionadas 5 novas categorias (antes só existia "Saúde e Bem-Estar"), validadas com pedidos
reais ao catálogo `geo.turismodeportugal.pt/server/rest/services/TDP`:
- `accommodation` — SIGTUR_AlojamentosTuristicos/2 ("ET Existentes")
- `nature` — SIGTUR_TurismoNatureza/9 ("Centros BTT")
- `culture` — OpenData_AldeiasHistoricas/MapServer/1 ("Aldeias Históricas")
- `beaches-golf` — OpenData_PraiasBA/MapServer/30 + SIGTUR_Golfe/1
- `wine-tourism` — SIGTUR_Light_Enoturismo/1 ("Unidades de Enoturismo")

`SigturLayerConfig` passou a suportar `serverType` (`FeatureServer`/`MapServer`), necessário
porque alguns datasets OpenData só existem como MapServer. Mapeamento de campos alargado
(`Denominação`, `QUINTA`, `E_mail`, `Link`, `MORADA`, `Endereco_ent_gest`, etc.) por variarem
significativamente entre layers.

## Opção "Perto de mim" (2026-07-17)

Adicionada opção de filtrar pontos de interesse pela localização do utilizador:
- **Web**: botão "Perto de mim" usa `navigator.geolocation`, aplica `nearby` (raio 25km) ao
  pedido ArcGIS existente e ordena a lista por distância (haversine), mostrando um badge
  de distância em cada cartão. Segue o mesmo padrão já usado em `Hospitais.tsx`.
- **Mobile**: adicionada dependência `expo-location` (`~57.0.5`, compatível com Expo SDK 57)
  e plugin configurado em `app.json` com a permissão `locationWhenInUsePermission`. O tab
  Turismo passou a pedir permissão de localização em primeiro plano e a usar coordenadas
  reais do dispositivo; se a permissão for negada, mantém o fallback anterior (Lisboa).

## Paginação (2026-07-17)

Lista da app web (`Turismo.tsx`) passou a usar paginação de 20 itens por página, reutilizando
o componente `Pagination` já existente (mesmo padrão de `Hospitais.tsx`). A página reinicia
para 1 sempre que a pesquisa, categoria ou modo "Perto de mim" mudam.

## Correção definitiva de CORS: JSONP (2026-07-17)

O workaround anterior via `corsproxy.io` deixou de funcionar em produção
(`net::ERR_SSL_VERSION_OR_CIPHER_MISMATCH` reportado pelo utilizador; confirmado via curl que
o serviço agora responde 403 "Server-side requests are not allowed on your plan" — passou a
exigir plano pago). Testadas alternativas gratuitas (`allorigins.win`, `api.codetabs.com`):
ambas falham com 522/520 a alcançar `geo.turismodeportugal.pt` a partir de infraestrutura cloud.

Solução adotada: **JSONP nativo do ArcGIS REST** (parâmetro `f=json&callback=...`), que
contorna CORS por completo porque o pedido é feito via `<script>`, não via `fetch`/XHR — não
depende de nenhum proxy de terceiros. Verificado manualmente com sucesso em todas as layers
configuradas (FeatureServer e MapServer).

Alterações:
- `packages/core/src/api/sigtur.ts`: `configureSigturRequestUrl` (transformação de URL)
  substituído por `configureSigturTransport` (função de transporte pluggable). Transporte por
  omissão continua a ser `fetch` direto com `f=geojson` (usado pela app mobile).
- `apps/web/src/lib/sigtur.ts`: reescrito para implementar o transporte via JSONP
  (`<script>` dinâmico + callback global), convertendo a resposta Esri JSON (`x`/`y`,
  `attributes`) para o formato GeoJSON esperado pelo normalizador.
- Removido o proxy `/api/sigtur` do `vite.config.ts` (deixou de ser necessário).

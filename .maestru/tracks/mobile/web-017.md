# WEB-017: Carris: migrar apps (web/mobile/android-auto) para o `.sqlite` local (dados estáticos)

## Contexto

Passo final da estratégia (revista): `apps/web/src/hooks/useCarris.ts` (e equivalentes em `apps/mobile`, `apps/android-auto`) deixam de pedir por rede as entidades **estáticas** (linhas, paragens, rotas, patterns, shapes, operadores) e passam a consultar o `.sqlite` local via WASM (WEB-010/011/012/016). Os dados **dinâmicos** (chegadas, veículos, alertas — WEB-013/014/015) continuam a ser pedidos em tempo real às APIs oficiais, exatamente como hoje.

**Validação:** risco principal é regressão de UX — `useCarris.ts` já tem lógica fina (fallback v1/v2, cálculo de distância haversine, normalização `V1Stop`→`CMStop`). Ao migrar, manter os mesmos contratos de tipos (`CMLine`, `CMStop`, `CMVehicle`, `CMRealtime`) para minimizar alterações nos componentes consumidores (`Transportes.tsx`). Rever `vite.config.ts` (proxy `/api/carris` para `gateway.carris.pt`) — provavelmente obsoleto/a remover.

## Objetivo

- Trocar `useCarrisLines`/`useCarrisStops`/`useCarrisLinePatterns` (dados estáticos) para lerem do `.sqlite` local (WASM) em vez de `fetch` à API.
- Manter `useCarrisVehicles`/`useStopRealtime`/alertas como fetch direto (sem alterações de fundo, só reaproveitar tipos).
- Remover proxy antigo de `vite.config.ts` se deixar de ser necessário.
- Validar paridade funcional (linhas, paragens, veículos, chegadas) e medir ganho de performance/offline antes de remover os `fetch` estáticos antigos.
